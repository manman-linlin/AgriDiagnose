# 多模态大模型智慧农业系统详细设计文档

## 1. 项目概述

### 1.1 项目名称
基于图像识别与大模型 Agent 的农作物病虫害诊断系统设计与实现

### 1.2 项目背景
农作物病虫害会直接影响作物产量、品质和农户收益。传统识别方式主要依赖人工经验，存在识别效率低、知识门槛高、难以规模化推广等问题。随着深度学习、视觉识别和大语言模型技术的发展，可以构建“图像识别 + 智能决策”的农业辅助诊断系统，实现病虫害自动识别与防治建议生成。

本项目面向智慧农业场景，设计一个可交互的 Web 系统。用户上传农作物叶片图片后，系统先通过图像分类模型识别病虫害类别，再将识别结果与图片信息传递给大模型 Agent，由 Agent 生成防治方案、风险提示与辅助建议，最终通过可视化页面展示诊断结果。

### 1.3 项目目标
1. 实现农作物叶片图片上传、预览与诊断功能。
2. 训练或集成一个病虫害图像识别模型。
3. 调用大模型 API 生成病虫害防治建议。
4. 构建前后端分离的 Web 应用系统。
5. 提供识别结果、置信度、建议内容及历史记录的可视化展示。
6. 完成项目报告、测试验证与演示材料。

### 1.4 项目定位
本项目偏软件开发实现，强调系统完整性与可演示性。深度学习模型用于核心识别能力，大模型用于决策生成与语言表达，系统实现侧重于工程集成、交互体验与可视化展示。

## 2. 需求分析

### 2.1 功能需求
系统需满足以下功能：

1. 用户可以上传农作物叶片图片。
2. 系统可以对图片进行预处理并调用图像分类模型。
3. 系统可以输出病虫害类别及置信度。
4. 系统可以将识别结果传递给大模型接口。
5. 大模型可以生成病害说明、防治措施和注意事项。
6. 系统可以展示识别结果和防治方案。
7. 系统可以记录每次诊断历史。
8. 系统可以提供简单统计图表，如类别分布、历史次数等。

### 2.2 非功能需求
1. 响应时间合理，单次诊断结果应在可接受时间内返回。
2. 系统界面简洁，适合课堂展示和答辩演示。
3. 模块之间解耦，便于后续替换模型或大模型接口。
4. 结果输出稳定，避免大模型输出格式混乱。
5. 支持基础错误处理，如图片格式错误、接口调用失败等。

### 2.3 约束条件
1. 大模型采用 API 调用，不训练基础大模型。
2. 图像识别模型以轻量化、易训练、易部署为主。
3. 课程设计周期有限，优先保证系统可运行和可展示。

## 3. 总体设计

### 3.1 系统架构
系统采用前后端分离架构，整体分为四层：

1. 前端展示层：负责图片上传、结果显示、历史记录与图表展示。
2. 后端服务层：负责业务编排、接口转发和结果聚合。
3. 模型推理层：负责病虫害分类预测。
4. 大模型决策层：负责生成防治建议和辅助说明。

### 3.2 系统流程
1. 用户在前端上传农作物叶片图片。
2. 后端接收图片并保存到本地或临时目录。
3. 图像识别模型对图片进行推理，得到病虫害类别与置信度。
4. 后端将识别结果、作物信息和必要上下文发送给大模型 API。
5. 大模型 Agent 根据输入生成防治方案。
6. 后端将识别结果和生成内容统一封装为 JSON 返回前端。
7. 前端展示结果，并将记录保存到历史列表中。

### 3.3 总体模块划分
1. 用户交互模块
2. 图像预处理模块
3. 图像识别模块
4. 大模型 Agent 模块
5. 历史记录模块
6. 可视化展示模块
7. 系统管理模块

## 4. 模块详细设计

### 4.1 用户交互模块
#### 功能说明
该模块提供用户界面入口，支持图片上传、识别触发、结果查看与历史记录浏览。

#### 主要页面
1. 首页/诊断页
2. 历史记录页
3. 统计分析页
4. 系统配置页

#### 交互流程
1. 用户选择图片。
2. 页面显示图片预览。
3. 用户点击“开始识别”。
4. 页面展示加载状态。
5. 接收诊断结果后渲染病害类别、防治建议和置信度。

### 4.2 图像预处理模块
#### 功能说明
对输入图片进行标准化处理，保证模型输入一致性。

#### 处理内容
1. 图片尺寸缩放。
2. 像素归一化。
3. RGB 通道转换。
4. 张量化处理。

#### 设计说明
预处理流程应与训练时保持一致，避免训练-推理分布不一致导致性能下降。

### 4.3 图像识别模块
#### 功能说明
负责对叶片图片进行病虫害分类，输出类别和置信度。

#### 模型选择建议
1. ResNet18
2. MobileNetV3
3. EfficientNet-B0
4. MobileViT

#### 推荐方案
课程设计优先推荐 `ResNet18` 或 `MobileNetV3`，原因如下：
1. 训练稳定。
2. 代码实现简单。
3. 模型体积适中。
4. 适合后续部署与演示。

#### 输出结果
```json
{
  "label": "番茄早疫病",
  "confidence": 0.92,
  "top3": [
    {"label": "番茄早疫病", "score": 0.92},
    {"label": "番茄晚疫病", "score": 0.05},
    {"label": "番茄叶霉病", "score": 0.03}
  ]
}
```

### 4.4 大模型 Agent 模块
#### 功能说明
根据图像识别结果，调用大模型 API 生成病虫害诊断说明与防治方案。

#### 输入内容
1. 作物名称
2. 病虫害类别
3. 识别置信度
4. 图片简要描述
5. 知识上下文或提示词模板

#### 输出内容
1. 病害名称解释
2. 常见症状
3. 可能成因
4. 防治建议
5. 风险提示
6. 是否建议人工复核

#### Agent 设计思路
Agent 不直接“自由发挥”，而是通过结构化提示词限制输出格式，确保结果可控、便于前端展示。

#### 推荐输出格式
```json
{
  "disease_name": "番茄早疫病",
  "symptoms": "叶片出现褐色病斑，边缘清晰，严重时扩展融合。",
  "cause": "多由真菌感染引起，常在高温高湿条件下发生。",
  "prevention": [
    "及时清理病叶病株",
    "保持通风透光",
    "合理轮作与密植"
  ],
  "treatment": [
    "在专业指导下选择合规药剂",
    "注意安全间隔期",
    "结合农业防治和化学防治"
  ],
  "risk_level": "中",
  "manual_check_required": false
}
```

#### 置信度策略
1. 当分类置信度高于阈值时，正常生成防治建议。
2. 当分类置信度低于阈值时，提示结果仅供参考，建议人工复核。

### 4.5 历史记录模块
#### 功能说明
记录每次诊断的时间、图片路径、识别类别、置信度和防治结果。

#### 数据字段
1. 记录编号
2. 图片名称
3. 识别结果
4. 置信度
5. 防治建议摘要
6. 诊断时间

#### 使用场景
1. 查询最近诊断结果。
2. 回看某次识别记录。
3. 统计高频病害类别。

### 4.6 可视化展示模块
#### 功能说明
用于增强系统表现力和答辩展示效果。

#### 展示内容
1. 病虫害类别标签
2. 置信度进度条
3. 防治建议卡片
4. 历史诊断列表
5. 分类统计图表

#### 图表建议
1. 柱状图：各病害识别次数
2. 饼图：历史记录类别占比
3. 折线图：识别任务数量变化

### 4.7 系统管理模块
#### 功能说明
负责配置模型路径、API Key、类别标签、阈值等系统参数。

#### 配置项
1. 模型权重路径
2. 输入图片尺寸
3. 识别阈值
4. 大模型 API 地址
5. 大模型密钥
6. 结果保存目录

## 8. 关键业务逻辑

### 8.1 识别主流程
1. 用户上传图片。
2. 后端保存图片并做预处理。
3. 模型输出病害类别和置信度。
4. 后端判断置信度是否达标。
5. 若达标，则调用大模型生成建议。
6. 若不达标，则提示结果仅供参考。

### 8.2 提示词生成逻辑
为提高大模型回答质量，系统应采用固定模板构建提示词，例如：

```text
你是一名农业病虫害助手。请根据以下信息生成结构化诊断建议：
作物名称：...
病害类别：...
识别置信度：...
请输出症状、成因、防治建议、风险提示，并使用 JSON 格式返回。
```

### 8.3 异常处理逻辑
1. 图片格式不支持时，返回错误提示。
2. 模型推理失败时，返回默认错误信息。
3. 大模型接口超时或失败时，仅展示识别结果。
4. 历史记录保存失败时，不影响主诊断流程。

## 12. 风险分析

### 12.1 技术风险
1. 图像识别准确率不足。
2. 大模型输出内容不稳定。
3. 前后端接口联调复杂。

### 12.2 应对策略
1. 选择成熟轻量模型，保证基础效果。
2. 使用结构化提示词约束大模型输出。
3. 先完成最小可用版本，再逐步扩展功能。

### 14.5 与其他模块的接口边界

```
Web 前端
  │
  ├──→ 3.Web/Backend（FastAPI）    ← 唯一运行时依赖
  │     ├── /api/predict           → 调用 ConvNeXt-Tiny 推理
  │     ├── /api/chat/*            → 调用 4.Agent 豆包 API
  │     ├── /api/encyclopedia/*    → 读写 2.Model/encyclopedia.json
  │     └── /api/contribute/*      → 写入 2.Model/contributed/
  │
  ├──→ 2.Model/Data/class_labels.json  ← 静态引用（编译时）
  ├──→ 5.Test/image/                    ← 百科图片复用
  │
  └── 不直接依赖 4.Agent/（通过 /api/chat 后端中转）
```

> **关键设计原则**：Web 前端不直接调用模型推理或豆包 API。所有外部调用均通过 FastAPI 后端中转，前端只关心 HTTP 接口契约（URL + 请求体 + 响应 JSON 格式）。

### 14.7 视觉特效技术说明

| 特效 | 技术 | 性能注意 |
|------|------|---------|
| 玻璃拟态 | `backdrop-filter: blur()` | 移动端降级为纯色半透明 |
| Mesh Gradient 背景 | 5 色块 radial-gradient + `blur(50px)` + `translate3d(0,0,0)` GPU 合成层 | 移动端 blur 降至 40px |
| 3D 卡片透视 | `perspective: 800px` + hover `rotateX/Y` | 移动端降级为 `translateY(-2px)` |
| 浮动光斑 | `@keyframes` + `will-change: transform` | `prefers-reduced-motion` 时禁用 |
| 数字滚动 | `requestAnimationFrame` + easeOutCubic | 仅触发一次，不影响持续性能 |
| 逐字输出 | `setInterval` 30-55ms | 在 `beforeUnmount` 清理定时器 |
| Skeleton 骨架屏 | shimmer 渐变动画 | 加载完成后移除 DOM |
