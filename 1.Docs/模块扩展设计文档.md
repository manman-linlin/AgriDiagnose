## 1. 总体概述

### 1.1 当前系统回顾

现有系统已完成：图片上传 → ConvNeXt-Tiny 分类 → 返回中英文病害名 + Top-3 置信度 → 历史记录 → 统计图表。

### 1.2 本次扩展目标

| 模块 | 新增内容 | 解决的核心问题 |
|------|----------|---------------|
| Agent 对话 | 多模态 LLM 对话页面、纠错、防治建议 | 模型训练数据为实验室拍摄，真实场景可能有偏差；用户需要可交互的防治指导 |
| 数据收集 | 用户贡献图片 + 标签上传页面 | 38类覆盖有限，需要众包数据实现模型持续迭代 |
| 病害百科 | 病害图鉴浏览、作物筛选、知识沉淀 | 用户不上传图片也能了解病害知识；AI 生成的防治方案可沉淀为结构化知识库 |
| 页面优化 | 全新 UI/UX 设计 | 当前单页三页签不够用，需升级为侧边栏多页面布局 |

## 2. 模块一：Agent 智能对话模块

### 2.1 功能定位

将分类结果 + 原始叶片图片发送给豆包多模态大模型，实现：

- **多模态纠错**：豆包直接"看"图片，结合分类结果给出二次判断（实验室数据 vs 真实场景）
- **防治建议对话**：用户可与 AI 多轮对话，追问具体用药、用量、时机等
- **结构化输出**：防治方案包含症状/成因/预防/治疗/风险等级，前端卡片式展示

### 2.2 页面设计 —— AI 诊断对话页

#### 2.2.1 页面入口

从诊断结果页点击 **"🤖 AI 深度解读"** 按钮，携带当前分类结果 + 图片跳转到对话页。

#### 2.2.5 纠错机制设计

由于模型在 PlantVillage（实验室环境）上训练，真实场景图片可能存在分布差异（光线、背景、拍摄角度等），豆包多模态能力可以辅助判断：

| 场景 | 模型置信度 | AI 判断 | 系统行为 |
|------|-----------|---------|---------|
| 模型+AI 一致 | ≥80% | 一致 | 绿色确认标记，正常出方案 |
| 模型+AI 一致 | <80% | 一致 | 黄色提示"建议人工复核"，仍出方案 |
| AI 质疑模型结果 | 任意 | 不一致 | 橙色警告，展示双方判断，建议以AI为准 |
| AI 无法判断 | 任意 | 不确定 | 灰色提示"图片质量不足以判断，请重新拍摄" |

**Prompt 设计要点：**

```text
[System]
你是一名拥有 20 年经验的农业植物病理学专家，同时具备计算机视觉分析能力。
你会同时收到一张农作物叶片图片和一个来自图像分类模型的预测结果。
请完成以下任务：

1. **多模态复核**：仔细观察叶片图片，判断模型分类是否正确。
   - 如果正确，说明你看到了哪些支持该判断的视觉特征
   - 如果存疑，指出视觉上的疑点和你倾向的判断
   - 如果图片质量太差无法判断，诚实说明

2. **结构化防治方案**（无论是否纠正分类，都需给出）：
   按以下 JSON 格式输出：
   {
     "review": {
       "agrees_with_model": true/false,
       "ai_diagnosis": "你的判断",
       "visual_evidence": ["观察到的特征1", "特征2"],
       "confidence_note": "关于图片质量的说明"
     },
     "advice": {
       "disease_name": "病害名",
       "symptoms": "症状描述",
       "cause": "发病原因",
       "prevention": ["措施1", "措施2"],
       "treatment": ["方法1", "方法2"],
       "risk_level": "高/中/低",
       "manual_check_required": true/false
     }
   }

3. **对话引导**：在回复末尾，提供 2-3 个用户可能关心的追问方向。

注意事项：
- 药剂建议使用通用名，提醒安全间隔期
- 图片来自真实田间拍摄，可能与实验室标准照有差异
- 如不确定，宁可建议人工复核，不要给出错误指导
```

#### 2.3.2 `4.Agent/` 模块设计

##### `4.Agent/config.py` — API 配置

需实现的功能：

- 定义豆包 API 的 Endpoint URL（`https://ark.cn-beijing.volces.com/api/v3/chat/completions`）
- 定义默认模型名称（`doubao-pro-32k`，支持多模态图片输入）
- 从环境变量 `DOUBAO_API_KEY` 读取 API Key，避免硬编码
- 定义默认请求参数：`temperature=0.3`、`max_tokens=2048`、超时时间 60 秒
- 提供 `MODEL_CONFIG` 字典，方便后续切换模型或调整参数

##### `4.Agent/prompt.py` — Prompt 模板管理

需实现的功能：

- **System Prompt 常量**：定义农业病虫害专家的系统角色提示词（内容参见 2.2.5 节的 Prompt 设计），要求模型输出包含 `review`（多模态复核）和 `advice`（防治方案）的结构化 JSON
- **首轮消息构建函数** `build_first_message(image_base64, classification_result)`：
  - 接收图片的 Base64 编码和模型分类结果（Top-1 + Top-3）
  - 返回豆包多模态 API 要求的 messages 格式（`content` 数组同时包含 `image_url` 和 `text`）
  - text 中附带说明：图片来自田间拍摄，实验室训练数据可能有分布差异
- **追问消息构建函数** `build_followup_message(user_text)`：
  - 接收用户追问文本，返回标准 `{"role": "user", "content": ...}` 格式
  - 追问不带图片（节省 Token），仅首轮使用多模态
- **响应解析函数** `parse_ai_response(full_text)`：
  - 从 AI 返回的文本中提取 JSON 块（处理 AI 可能包裹的 markdown 代码块 ` ```json `）
  - 解析出 `review`（纠错结果）和 `advice`（防治方案）两个子对象
  - 解析失败时返回降级结构（将全文作为纯文本建议）

##### `4.Agent/agent.py` — 豆包 API 调用封装

需实现的功能：

- **会话数据类 `ConversationSession`**：
  - 字段：`session_id`、`image_base64`（首轮图片）、`classification_result`（模型分类结果）、`messages`（完整对话历史列表）
  - 每次新诊断创建一个会话，会话之间相互隔离
- **Agent 类 `DoubaoAgent`**：
  - 初始化：接收 `api_key` 和 `model` 参数，维护 `_sessions` 字典（内存级会话存储）
  - **`create_session(session_id, image_base64, classification)`**：
    - 创建新会话并存入 `_sessions` 字典
    - 会话 ID 由后端生成传入（UUID 格式），便于前后端关联
  - **`get_session(session_id)`**：获取已有会话，不存在时返回 None
  - **`async chat_stream(session_id, user_message=None)`**：
    - 异步生成器，逐 Token yield 返回（用于 SSE 流式传输）
    - 首次调用（无 `user_message`）：组装 System Prompt + 多模态首轮消息（图片+分类结果），发起流式请求
    - 后续调用：追加用户追问消息，发起纯文本流式请求
    - 使用 `httpx.AsyncClient.stream()` 发送 POST 请求，解析 SSE 响应（`data: {json}` 格式，`[DONE]` 结束）
    - 流式响应结束后将完整 AI 回复追加到会话历史
    - 异常处理：API 超时/网络错误时 yield 错误提示文本
  - **`delete_session(session_id)`**：释放会话（可选，用于清理过期会话）
- **模块级单例**：`agent = DoubaoAgent(api_key=config.API_KEY)`，Web 后端直接引用

#### 2.3.3 Web 后端新增路由

##### `3.Web/Backend/app/services/agent_service.py` — 薄封装层

该文件是 Web 后端与 `4.Agent/` 之间的桥梁，不包含豆包 API 调用逻辑，仅负责：

- 从 `4.Agent.agent` 导入单例 `agent` 对象
- 从 `4.Agent.config` 导入配置
- （可选）封装图片文件 → Base64 转换的工具函数

##### `3.Web/Backend/app/main.py` — 新增路由

需新增两个接口（修改 `main.py`）：

- **`POST /api/chat/start`**：
  1. 接收上传的叶片图片文件
  2. 调用已有的 `classifier.predict()` 执行模型分类，得到 Top-3 结果
  3. 将图片转为 Base64 字符串
  4. 调用 `agent.create_session()` 创建新会话
  5. 返回 `session_id` + 分类结果给前端

- **`GET /api/chat/stream/{session_id}?message=xxx`**：
  1. 根据 `session_id` 获取会话
  2. 调用 `agent.chat_stream()` 获取异步生成器
  3. 包装为 `StreamingResponse`（`text/event-stream`），每段数据以 `data: {json}\n\n` 格式发送
  4. 首次请求不带 `message` 参数，触发多模态分析；后续请求带 `message` 参数，进行纯文本追问

#### 2.4.2 SSE 流式接收

```javascript
// Vue3 组件中
async function startChat(sessionId) {
    const url = `/api/chat/stream/${sessionId}`;
    const eventSource = new EventSource(url);

    eventSource.onmessage = (event) => {
        if (event.data === '[DONE]') {
            eventSource.close();
            loading = false;
            return;
        }
        const { token } = JSON.parse(event.data);
        // 逐字追加到当前 AI 消息气泡
        currentAiMessage += token;
    };

    eventSource.onerror = () => {
        eventSource.close();
        error.value = '连接中断，请重试';
    };
}
```

> **注意**：原生 `EventSource` 不支持 POST，首次会话创建用 `POST /api/chat/start`，后续流式用 `GET /api/chat/stream/{session_id}?message=xxx`（GET 请求参数需 URL Encode）。

#### 3.3.3 `collector.py` — 数据收集服务

##### 数据模型定义

- **贡献模式枚举 `ContributionMode`**：`EXTEND`（扩展已有类别）、`NEW`（新增病害类型）
- **审核状态枚举 `ReviewStatus`**：`PENDING`（待审核）、`APPROVED`（已采纳）、`REJECTED`（未通过）
- **贡献记录数据类 `Contribution`**：
  - 基础字段：`id`（UUID）、`mode`（提交模式）、`submit_time`（提交时间）、`status`（审核状态）
  - 扩展模式字段：`existing_class`（对应的英文类名，如 `Tomato___Early_blight`）
  - 新增模式字段：`crop_name`（作物名）、`disease_name`（病害名）、`disease_description`（病害描述）
  - 图片字段：`image_count`、`image_paths`（相对路径列表）
  - 可选补充字段：`location`（拍摄地点）、`photo_date`（拍摄时间）、`notes`（备注）
  - 审核字段：`review_notes`（审核备注）

##### `CollectorService` 类需实现的方法

- **`submit(mode, images, ...)` → Contribution**：
  1. 生成唯一贡献 ID（UUID 前 12 位）
  2. 扩展模式：图片存入 `contributed/extend/{英文类名}/`，文件命名格式 `ext_{类名简写}_{时间戳}_{序号}.jpg`
  3. 新增模式：图片存入 `contributed/new/{作物}___{病害}/`，同时写入 `metadata.json` 记录作物名、病害名、描述和提交时间
  4. 图片统一转为 RGB JPEG 格式保存，质量 92
  5. 构建 `Contribution` 记录写入 `contributions.json` 索引
  6. 返回贡献记录对象

- **`list_contributions(status=None)` → list[dict]**：
  - 读取 `contributions.json` 索引文件
  - 支持按审核状态筛选（`pending` / `approved` / `rejected`）
  - 按提交时间倒序返回

- **`review(contribution_id, approved, notes=None)`**：
  - 管理员审核接口：将指定贡献记录的状态改为 `approved` 或 `rejected`
  - 支持附加审核备注

- **`get_stats()` → dict**：
  - 返回统计信息：总提交数、总图片数、已采纳图片数、扩展/新增提交数、待审核数

##### JSON 索引文件读写

- 读取 `contributions.json` 时处理文件不存在的情况（返回空列表）
- 写入时使用 `ensure_ascii=False` + `indent=2` 保证中文可读
- 图片保存失败时不写入索引

#### 3.3.4 `main.py` — 新增路由

需在 `main.py` 中新增以下 4 个接口：

- **`POST /api/contribute`**：
  - 接收 `mode`（`extend` / `new`）+ 多张图片文件 + 标注参数
  - 参数校验：模式合法性、扩展模式必填 `existing_class`、新增模式必填 `crop_name` + `disease_name`
  - 过滤非图片文件，至少保留一张有效图片
  - 调用 `collector_service.submit()` 完成提交
  - 成功返回 `{"code": 200, "message": "提交成功，感谢您的贡献！", "data": {"id": "..."}}`

- **`GET /api/contribute/list?status=pending`**：
  - 返回贡献记录列表，支持按审核状态筛选

- **`GET /api/contribute/stats`**：
  - 返回贡献统计数据（总提交/图片数/待审核数等）

- **`GET /api/contribute/classes`**：
  - 读取 `class_labels.json` 返回已有 38 类的中英文对照列表
  - 供前端扩展模式下的病害下拉选择器使用

#### 4.4.2 `encyclopedia.py` — 百科服务

##### `EncyclopediaService` 类需实现的方法

- **`load()`**：
  - 服务启动时从 `encyclopedia.json` 加载全部百科条目到内存
  - 构建按作物分组、按名称索引的查找结构，加速查询

- **`list_all(crop=None, keyword=None, category=None)` → list[dict]**：
  - 返回百科条目摘要列表（不含完整症状描述和防治方案正文，仅保留摘要字段）
  - 支持按作物筛选（`crop=番茄`）
  - 支持关键词搜索（匹配中文名、英文名、症状关键词）
  - 支持按病害类型筛选（真菌性/细菌性/病毒性/虫害/健康）
  - 排序：同一作物按风险等级从高到低排列

- **`get_detail(disease_id)` → dict**：
  - 返回完整百科条目（包含症状、发病规律、防治方案全部字段）
  - 未找到时返回 None

- **`get_crops()` → list[dict]**：
  - 返回作物列表，每项包含作物名 + 该作物下的病害数 + 图标emoji
  - 用于前端侧栏筛选

- **`get_categories()` → list[str]**：
  - 返回病害分类列表（真菌性病害、细菌性病害、病毒性病害、虫害、健康）

- **`get_similar_diseases(disease_id)` → list[dict]**：
  - 返回易混淆病害列表（来自百科条目的 `similar_diseases` 字段）

- **`refresh_from_agent_data()`**：
  - （可选）从 AI 对话历史中提取高质量防治方案
  - 对比百科现有内容，标记可更新的字段
  - 需要管理员审核后才能合并到正式百科数据

#### 4.4.3 `main.py` — 新增路由

需新增以下接口：

- **`GET /api/encyclopedia/list?crop=番茄&keyword=早疫&category=真菌性`**：
  - 返回筛选后的病害摘要列表
  - 所有参数可选，无参数时返回全部 38 类
  - 响应格式：`{"code": 200, "data": [...], "total": 38}`

- **`GET /api/encyclopedia/{disease_id}`**：
  - 返回病害完整详情
  - 未找到返回 404

- **`GET /api/encyclopedia/crops`**：
  - 返回作物分组列表（供前端侧栏渲染）

- **`GET /api/encyclopedia/categories`**：
  - 返回病害分类列表（供筛选标签）

### 4.6 百科数据建设策略

百科的**文字内容**（症状、发病规律、防治方案等）由 AI 从网络收集并整理，图片则复用 `5.Test/image/` 中已有的测试图片。数据建设分为三个阶段：

| 阶段 | 内容 | 数据来源 | 方式 |
|------|------|----------|------|
| 第一阶段 | 38 类基础数据 | AI 网络搜索 + 公开农技资料 | AI 自动搜索→整理为结构化 JSON→人工审核校对 |
| 第二阶段 | 内容增强 | AI 对话中生成的结构化方案 | 标记高质量回复 → 人工审核入库 |
| 第三阶段 | 用户贡献病害 | 数据收集模块中审核通过的新病害 | 管理员编写百科条目 |

**第一阶段 AI 收集流程：**

```
对每类病害：
  1. AI 搜索（WebSearch）中文病害名 + "症状 防治 发病规律"
  2. AI 从搜索结果中提取结构化信息：
     - 典型症状（列表）
     - 病原菌名称（中英文 + 拉丁学名）
     - 发病条件（温度、湿度、季节）
     - 传播途径
     - 预防措施（农业防治 + 物理防治）
     - 化学防治（常见药剂 + 使用方法 + 安全间隔期）
     - 易混淆病害及区分要点
  3. AI 将提取结果填充到百科 Schema JSON 模板
  4. 人工审核：核对关键信息准确性（药剂名、安全间隔期等）
  5. 通过后写入 encyclopedia.json
```

> 第一阶段在开发期间完成（作为百科的初始内容），第二、三阶段随系统使用持续迭代。
>
> **图片来源**：百科图片直接复用 `5.Test/image/` 中的 38 类测试图片（每类 5 张），不做额外收集。详见 [4.3.2 节](#432-数据存储) 的目录对照表。

## 7. 数据库设计

### 7.1 从 JSON 文件迁移到 SQLite

当前使用 `history.json` + `contributions.json` 两个 JSON 文件，扩展后数据量增大，建议迁移到 SQLite。

```sql
-- 诊断历史表
CREATE TABLE diagnoses (
    id TEXT PRIMARY KEY,
    filename TEXT NOT NULL,
    image_path TEXT NOT NULL,
    label_en TEXT NOT NULL,
    label_cn TEXT NOT NULL,
    crop TEXT,
    disease TEXT,
    confidence REAL NOT NULL,
    top3_json TEXT,           -- JSON: Top-3 完整结果
    created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime'))
);

-- AI 对话会话表
CREATE TABLE chat_sessions (
    id TEXT PRIMARY KEY,
    diagnosis_id TEXT,         -- 关联的诊断记录
    image_path TEXT,
    model_agrees INTEGER,      -- AI是否同意模型判断 (0/1/NULL)
    messages_json TEXT,        -- 完整对话历史 JSON
    created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime')),
    FOREIGN KEY (diagnosis_id) REFERENCES diagnoses(id)
);

-- 数据贡献表
CREATE TABLE contributions (
    id TEXT PRIMARY KEY,
    mode TEXT NOT NULL CHECK (mode IN ('extend', 'new')),
    existing_class TEXT,
    crop_name TEXT,
    disease_name TEXT,
    disease_description TEXT,
    image_count INTEGER NOT NULL DEFAULT 0,
    image_paths_json TEXT,     -- JSON: 图片路径列表
    location TEXT,
    photo_date TEXT,
    notes TEXT,
    status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'rejected')),
    review_notes TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now', 'localtime'))
);
```

> **过渡策略**：短期内可继续使用 JSON 文件（课程设计数据量不大），但代码中抽象出 `StorageInterface`，后续可无缝切换 SQLite。

| `AppStore`（Vue.reactive） | `getApp().globalData` | 低 |
| `Api.request()`（fetch） | `wx.request` / `wx.uploadFile` | **低（只改 1 个函数）** |
| `<input type="file">` | `wx.chooseImage` | 中 |
| `EventSource`（SSE） | `wx.request enableChunked` | 中 |
| ECharts | `echarts-for-weixin`（分包） | 中 |
| CSS 变量 | WXSS 变量（完全支持） | 低 |
| CSS `px` | `rpx = px ÷ 375 × 750` | 低 |
| `backdrop-filter` | iOS 支持 / Android 降级 | 需降级 |
| 3D transforms | 降级为简单位移 | 需降级 |
